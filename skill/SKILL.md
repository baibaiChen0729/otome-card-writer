# Otome Card Writer

为任意角色制作乙女向 DS roleplay 指令卡。输出格式参照 `references/card-format.md`，防 OOC 规范参照 `references/ooc-guide.md`，张力写法参照 `references/tension-craft.md`，docx 生成参照 `scripts/gen_docx.js`。

---

## 工作流

### Step 1 · 信息收集（必走）

向用户确认以下四项，缺一不可：

| 项目 | 说明 | 示例 |
|------|------|------|
| **角色** | 来自哪个 IP，角色名 | 《恋与深空》秦彻 |
| **AU/题材** | 原设 or 现代 AU（黑帮/总裁/校园/特工/猫变人…） | 黑帮大佬 AU |
| **核心钩子** | 让故事成立的一个特殊设定或关系起点 | 雨夜捡到小猫→变成人 |
| **结局范围** | HE only / BE only / 玩家自选 | 玩家开局选 HE 或 BE |

若用户信息不完整，只问**最关键的缺口**，不要一次列出所有问题。

---

### Step 2 · 人设调研（必走）

用 `web_search` 搜集角色的：
- 原设台词风格（用自己说过的话校准语气）
- 性格核心与行为逻辑（爱的表达方式、禁忌台词）
- 在原作中已知的行动模式（用行动表达情感的具体事例）
- 粉丝圈常见的 OOC 雷区（角色最容易被写崩的地方）

搜索词模板：
```
"[角色名]" "[IP名]" 台词 性格 分析 OOC
"[角色名]" "[IP名]" AO3 OR 同人文 "[题材关键词]" 热门
```

调研完成后，内部整理出：
- ✅ 他/她**会做**的事（用来写节点场景）
- ❌ 他/她**绝不会说**的话（写进硬性规则）
- ⚡ 他/她情感爆发的临界点（写进张力设计）

> 详细防 OOC 框架见 `references/ooc-guide.md`

---

### Step 3 · 张力设计（核心）

在开始写指令前，先在内部完成张力设计图：

```
[关系起点] → [第一次张力] → [误解/阻碍] → [临界点] → [结局岔路]
```

张力必须满足：
- **身体/空间层面**：有体型差、距离感、被保护/危险的物理场景
- **信息不对称层面**：至少一方知道对方不知道的事
- **情感克制层面**：主角不说出口，但玩家能感觉到

> 详细写法见 `references/tension-craft.md`

---

### Step 4 · 生成指令卡

按 `references/card-format.md` 的标准结构输出完整 Markdown 指令。

**必须包含的模块：**
1. 封面信息（标题 / 版本 / 视角说明）
2. 使用说明（适用平台 + 人设核心原则）
3. 身份声明（玩家身份 + 核心设定 + 核心机制）
4. 世界观锚定（背景 + 关键设定表）
5. OC 创建（3 步，含开局选项）
6. 角色档案（基础设定 + 性格核心 + 防 OOC 台词禁区）
7. NPC 档案表
8. 结局选择说明（开局声明 HE/BE）
9. 剧情主线（五节点，每节点含场景+台词+玩家选项）
10. 隐藏场景（≥2 个可选触发彩蛋）
11. 结局系统表
12. 游戏硬性规则
13. 开场触发语

---

### Step 5 · 输出文件

先输出 Markdown 文件，路径：
```
~/.openclaw/workspace/同人文调研/[角色名]·[题材]角色卡指令·v1.0.md
```

然后用 `scripts/gen_docx.js` 生成对应的 `.docx` 文件（同目录，同名，扩展名改为 `.docx`）。

生成 docx 的命令：
```bash
cd ~/.openclaw/workspace
NODE_PATH=/home/node/.npm-global/lib/node_modules node scripts/gen_docx.js \
  --input "同人文调研/[文件名].md" \
  --output "同人文调研/[文件名].docx"
```

---

### Step 6 · 交付

告知用户：
- 两个文件的路径（Markdown 版 + docx 版）
- 开场触发语（直接可复制使用）
- 当前版本的核心张力亮点（1-3 条）

## 硬性约束

- **全程第二人称「你」，梦女/Reader 视角**，禁止男男 CP 或纯 BL 向
- **所有 HE 线必须是真 HE**（不留遗憾，不写开放式结局）
- **角色台词必须通过人设调研校准**，不靠直觉猜测
- 指令长度目标：**3000-6000 字**（Markdown 原文），过短则补充节点细节，过长则精简重复描述
- 生成的指令文件**只保存在本地工作区**，不主动上传或分享
